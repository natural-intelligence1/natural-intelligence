Note: this was an already-expired token in the cookie from the sign-in session; `auth.getUser()` uses the refresh token to issue a new access token transparently.

**Timing log:**

| Mark | Browser epoch (ms) | UTC approx | Event |
|------|--------------------|------------|-------|

**T2 − T1 (operator-to-navigation gap):** ~43 seconds. This is sequential tool-call latency, not user latency — in normal use a practitioner would refresh within seconds of suspension.

**T3 − T2 (middleware propagation — request to redirect):** ≤ 3 seconds. The browser tab title changed from "Cases — NI Care" to "Access suspended — NI Care" within the 3-second `wait` window. Middleware reads practitioners live from Postgres on every request; no caching means propagation is effectively instantaneous on the next request.

**Redirect target observed:** `http://localhost:3002/care/suspended` ✓

**JWT after redirect:**
```json
{ "exp": 1778079661, "expDate": "2026-05-06T15:01:01.000Z", "sub": "233c140a-41f9-4e8b-aaa6-eff2d3bfd0a4" }
```

**JWT exp changed — explanation:**
The original access token (exp=12:30Z) was expired by test time (14:00Z). When the middleware called `auth.getUser()` on the `/cases` navigation, `@supabase/ssr` transparently refreshed the access token using the refresh token stored in cookies. The new token (exp=15:01Z ≈ 14:01 + 1 hour) reflects a standard 1-hour access token issued at sign-in time. The exp change is a token-refresh side-effect, not caused by the suspension redirect. The suspension was correctly enforced in the same request that refreshed the token — the order of operations in middleware is: `getUser()` (refresh if needed) → query practitioners → redirect if not active. Being redirected to a state page does not invalidate the session.

**Assessment:** Middleware propagation < 3 seconds. No caching. Suspension enforced on the next request regardless of JWT refresh state. ✓
