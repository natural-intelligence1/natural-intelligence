## Empirical test — suspension redirect (manual)

To confirm the suspension redirect completes within 10 seconds:

1. Sign in as an active practitioner (`status = 'active'`)
2. Confirm `/cases` loads normally
3. Run: `UPDATE practitioners SET status = 'suspended' WHERE id = '<practitioner-id>';`
4. Reload `/cases` within 10 seconds of the update
5. Expected: browser redirects to `/care/suspended`

Middleware performs no caching so the redirect takes effect on the next request. DB round-trip is a single `SELECT status FROM practitioners WHERE id = $1` — well within 10 seconds.

## Design decisions

- **Service-role client for practitioners lookup** — ensures status is always readable regardless of RLS policies on the practitioners table. `auth.getUser()` uses the anon key + user cookies (standard SSR pattern).
- **Broad matcher with explicit exemptions** — matcher catches all non-static routes; state pages are explicitly returned early in code so the exemption is visible and auditable.
- **Unauthenticated → `/`** — care app has no `/login` page; home is the appropriate landing for unauthenticated visitors.
- **78 lines excluding imports** — under the 80-line cap.
