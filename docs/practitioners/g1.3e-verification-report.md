## Issues Found and Corrected During G.1.3e

Before the twelve tests could pass, three files in `apps/care` were identified during an audit and corrected. These are documented here as part of the audit trail.

### Finding 1 — `apps/care/app/cases/page.tsx` (release-blocker)

**Problem:** The `/cases` page used `createAdminClient()` (service-role, bypasses RLS) to query `client_cases` with no practitioner filter. Every active practitioner who passed the middleware saw all cases in the database.

**Impact:** Violated the CRITICAL requirement of Test 6 — Practitioner B could access cases assigned only to Practitioner A. The V.6 anti-pattern (service-role in apps/care with no application-level enforcement) was reproduced in the cases list.

**Fix:** Replaced `createAdminClient()` with `createServerSupabaseClient()` (anon+cookies SSR client). The existing `case_practitioner_select` RLS policy (`EXISTS` check on `case_practitioner_work` where `practitioner_id = auth.uid()` and `status IN ('assigned', 'in_review')`) now scopes results automatically.

**Commit:** `98cb4d3` — `fix(care): use authenticated session + RLS for cases list`

---

### Finding 2 — `apps/care/app/cases/[caseId]/reasoning/page.tsx` (release-blocker)

**Problem:** Same class of bug. The reasoning page used `createAdminClient()` for both the `client_cases` load and the `getPractitionerTrace()` call. Any active practitioner could navigate directly to `/cases/<arbitrary-caseId>/reasoning` and view reasoning traces for cases they were not assigned to.

**Impact:** Same V.6 anti-pattern; data access violation — case UUIDs are guessable if leaked via browser history or URL sharing.

**Fix:** Replaced `createAdminClient()` with `createServerSupabaseClient()` for both queries. When RLS returns zero rows for an unassigned case, `.single()` returns a PGRST116 error → existing `if (caseErr || !clientCase) return notFound()` check fires → 404. No differential-response information leak (identical page for "case not found" and "case not accessible").

`getPractitionerTrace` type signature: `SupabaseClient<Database>` — compatible with the SSR client. No helper changes needed.

**Commit:** `89e7954` — `fix(care): use authenticated session + RLS for reasoning page; document legitimate service-role use in waitlist`

---

### Finding 3 — `apps/care/app/actions.ts` (no fix needed)

**Problem (audit):** `submitWaitlist` used `createAdminClient()`.

**Assessment:** Legitimate use. The endpoint is invoked anonymously (public waitlist form, `member_id: null`), the operation is append-only insert to `support_requests`, and there is no privacy boundary to enforce. No read-back. No alternative cleaner pattern without an explicit anon INSERT policy on `support_requests`.

**Action:** Added code comment documenting the rationale so future grep results require no re-derivation.
